# คำอธิบายคำสั่งโดยละเอียด

เอกสารนี้ให้รายละเอียดเกี่ยวกับ Slash Commands ทั้งหมดของ Novel Writer Skills

## การแบ่งหมวดหมู่คำสั่ง

### คำสั่งในระเบียบวิธีเจ็ดขั้นตอน

| คำสั่ง | หน้าที่ | ไฟล์ผลลัพธ์ | การพึ่งพา |
|------|------|---------|------|
| `/constitution` | สร้างรัฐธรรมนูญการเขียน | `.specify/memory/constitution.md` | ไม่มี |
| `/specify` | กำหนดข้อกำหนดของเรื่อง | `stories/[name]/specification.md` | `/constitution` |
| `/clarify` | ชี้แจงประเด็นที่คลุมเครือ | อัปเดต `specification.md` | `/specify` |
| `/plan` | จัดทำแผนการเขียน | `stories/[name]/creative-plan.md` | `/specify` |
| `/tasks` | แบ่งรายการงาน | `stories/[name]/tasks.md` | `/plan` |
| `/write` | ดำเนินการเขียนบท | `stories/[name]/content/chapter-XX.md` | `/tasks` |
| `/analyze` | วิเคราะห์ยืนยันคุณภาพ | รายงานการวิเคราะห์ (ในการสนทนา) | `/write` |

### คำสั่งติดตามและยืนยัน

| คำสั่ง | หน้าที่ | ไฟล์ผลลัพธ์ | ความถี่ |
|------|------|---------|------|
| `/track-init` | เริ่มต้นระบบติดตาม | `spec/tracking/*.json` | ใช้ครั้งแรก |
| `/track` | อัปเดตการติดตามแบบครบวงจร | อัปเดต `spec/tracking/*.json` | หลังแต่ละบท |
| `/plot-check` | ตรวจสอบความสอดคล้องของเนื้อเรื่อง | รายงานการวิเคราะห์ | ทุก 5-10 บท |
| `/timeline` | จัดการเส้นเวลา | `spec/tracking/timeline.json` | หลังเหตุการณ์สำคัญ |
| `/relations` | ติดตามความสัมพันธ์ของตัวละคร | `spec/tracking/relationships.json` | เมื่อความสัมพันธ์เปลี่ยน |
| `/world-check` | ยืนยันความสอดคล้องของโลก | รายงานการวิเคราะห์ | หลังจากตั้งค่าใหม่ |

## คำอธิบายโดยละเอียด

## `/constitution` - สร้างรัฐธรรมนูญการเขียน

### หน้าที่

สร้างหลักการและค่านิยมหลักสำหรับการเขียนนิยาย กำหนดเป็นไฟล์ "รัฐธรรมนูญ" ของการสร้างสรรค์

### สถานการณ์การใช้งาน

- เมื่อเริ่มต้นโครงการ
- เมื่อปรับเปลี่ยนทิศทางการสร้างสรรค์
- เมื่อมีผู้ร่วมงานใหม่

### ผลลัพธ์

สร้างไฟล์ `.specify/memory/constitution.md` ประกอบด้วย:
- ค่านิยมหลัก
- มาตรฐานคุณภาพ
- หลักการด้านรูปแบบ
- ข้อกำหนดด้านเนื้อหา

### ตัวอย่าง

```
/constitution
```

AI จะชักชวนให้คุณตอบคำถาม เช่น:
- "นิยายเรื่องนี้ต้องการสื่อแนวคิดหลักอะไร?"
- "อะไรคือสิ่งที่ไม่สามารถประนีประนอมได้?"
- "รูปแบบการเล่าเรื่องของคุณคืออะไร?"

### แนวทางปฏิบัติที่ดีที่สุด

- ใช้เวลา 15-20 นาทีในการคิดอย่างจริงจัง
- หลักการต้องชัดเจนและตรวจสอบได้
- ใช้คำศัพท์ที่ชัดเจน เช่น "ต้อง" "ห้าม"
- ทบทวนและอัปเดตเป็นประจำ (จัดการเวอร์ชัน)

---

## `/specify` - กำหนดข้อกำหนดของเรื่อง

### หน้าที่

กำหนดรายละเอียดต่าง ๆ ของเรื่องอย่างเช่นข้อกำหนดของผลิตภัณฑ์

### สถานการณ์การใช้งาน

- เมื่อมีความคิดใหม่สำหรับเรื่อง
- เมื่อต้องการรวบรวมความคิด
- ก่อนเริ่มเขียนอย่างเป็นทางการ

### ผลลัพธ์

สร้างไฟล์ `stories/[story-name]/specification.md` ประกอบด้วย:
- สรุปหนึ่งประโยค
- ภาพลักษณ์ของผู้อ่านกลุ่มเป้าหมาย
- จุดขายหลัก
- ตัวละครหลัก
- เกณฑ์ความสำเร็จ
- ข้อจำกัดและความเสี่ยง

### ตัวอย่าง

```
/specify
```

AI จะถาม:
- "อธิบายเรื่องนี้ด้วยหนึ่งประโยค"
- "ผู้อ่านกลุ่มเป้าหมายคือใคร?"
- "ความขัดแย้งหลักคืออะไร?"
- "จุดเด่นของเรื่องนี้คืออะไร?"

### เครื่องหมายพิเศษ

ในเอกสารข้อกำหนด ใช้:
- `[ต้องการชี้แจง]` - ทำเครื่องหมายจุดที่คลุมเครือ
- `[ความต้องการหลัก]` - ความต้องการที่ไม่สามารถประนีประนอมได้
- `[คุณสมบัติเสริม]` - เนื้อหาเพิ่มเติมที่ดีแต่ไม่จำเป็น

### แนวทางปฏิบัติที่ดีที่สุด

- เขียนร่างคร่าว ๆ ก่อน แล้วค่อยปรับปรุง
- ใช้หลัก SMART ในการกำหนดเกณฑ์ความสำเร็จ
- ระบุให้ชัดเจนว่า "สิ่งที่ไม่ใช่" และ "สิ่งที่เป็น"
- รักษาเอกสารให้กระชับ (ประมาณ 3-5 หน้า)

## `/clarify` - ชี้แจงประเด็นที่คลุมเครือ

### หน้าที่

ขจัดความคลุมเครือในข้อกำหนดด้วยคำถามที่แม่นยำ 5 ข้อ

### สถานการณ์การใช้งาน

- หลังจากทำ `/specify` เสร็จ
- เมื่อพบข้อกำหนดที่คลุมเครือ
- เมื่อสมาชิกในทีมมีความเข้าใจต่างกัน

### กระบวนการ

1. AI สแกน `specification.md`
2. ระบุจุดที่คลุมเครือหรือขัดแย้ง
3. สร้างคำถามสำคัญสูงสุด 5 ข้อ
4. อัปเดตข้อกำหนดโดยอัตโนมัติตามคำตอบ
5. เก็บประวัติการตัดสินใจ

### ตัวอย่าง

```
/clarify
```

คำถามที่เป็นไปได้:
- "แรงจูงใจหลักของตัวเอกคือการแก้แค้นหรือความยุติธรรม?"
- "ตอนจบเป็นแบบสุขนิยมหรือมีความเสียใจ?"
- "จังหวะของเรื่องเป็นแนวรวดเร็วเร้าใจ หรือค่อยเป็นค่อยไป?"

### แนวทางปฏิบัติที่ดีที่สุด

- ตอบอย่างตรงไปตรงมา อย่าตอบเพื่อให้ดู "ถูกต้อง"
- หากไม่รู้จริง ๆ ให้พูดว่า "ขอคิดก่อน"
- คำตอบมีผลต่อการเขียนในขั้นตอนต่อไป ควรระมัดระวัง
- สามารถรันซ้ำได้หลายครั้งจนกว่าจะชัดเจน

---

## `/plan` - จัดทำแผนการเขียน

### หน้าที่

แปลงข้อกำหนดเป็นแผนงานทางเทคนิคและโครงสร้างบท

### สถานการณ์การใช้งาน

- เมื่อข้อกำหนดชัดเจนแล้ว
- ก่อนเริ่มเขียน
- เมื่อต้องการวางแผนโครงสร้างเรื่องใหม่

### ผลลัพธ์

สร้างไฟล์ `stories/[story-name]/creative-plan.md` ประกอบด้วย:
- การเลือกวิธีการเขียน (สามองก์, การเดินทางของฮีโร่ ฯลฯ)
- การออกแบบโครงสร้างบท
- เครือข่ายความสัมพันธ์ของตัวละคร
- ระบบโลก
- เส้นเวลาและห่วงโซ่เหตุและผล
- แผนการวางและเก็บปม

### ตัวอย่าง

```
/plan
```

AI จะช่วยคุณ:
- เลือกวิธีการเขียนที่เหมาะสม
- ออกแบบโครงร่างบท
- วางแผนจังหวะและความตึงเครียด
- กำหนดเส้นทางการพัฒนาของตัวละคร

### แนวทางปฏิบัติที่ดีที่สุด

- แผนต้องเฉพาะเจาะจงแต่ไม่ละเอียดเกินไป
- เว้นที่ว่างสำหรับแรงบันดาลใจ
- เน้นวางแผน 1/3 ส่วนแรก
- ทบทวนและปรับแผนเป็นประจำ

---

## `/tasks` - แบ่งรายการงาน

### หน้าที่

แปลงแผนการเขียนเป็นงานที่สามารถดำเนินการได้

### สถานการณ์การใช้งาน

- เมื่อแผนเสร็จสมบูรณ์
- เมื่อต้องการเส้นทางการดำเนินการที่ชัดเจน
- เมื่อทำงานเป็นทีม

### ผลลัพธ์

สร้างไฟล์ `stories/[story-name]/tasks.md` ประกอบด้วย:
- งานเขียนบท
- งานปรับปรุงแฟ้มข้อมูลตัวละคร
- งานกำหนดโลก
- งานแก้ไขและขัดเกลา

### เครื่องหมายงาน

- `[P]` - สามารถดำเนินการพร้อมกันได้
- `[ขึ้นอยู่กับ:X]` - ขึ้นอยู่กับงาน X
- `[สูงสุด]` - ลำดับความสำคัญสูง
- `[P0/P1/P2]` - เครื่องหมายระดับความสำคัญ

### ตัวอย่าง

```
/tasks
```

สร้างผลลัพธ์เช่น:
```
[P0] ปรับปรุงแฟ้มข้อมูลตัวละครเอก
[P0][ขึ้นอยู่กับ:1] เขียนบทที่ 1: อารัมภบท
[P1] เพิ่มเติมการตั้งค่าโลกภูมิศาสตร์
[P1][P] เขียนบทที่ 2: การพบกันครั้งแรก
```

### แนวทางปฏิบัติที่ดีที่สุด

- ขนาดงานเหมาะสม (ใช้เวลา 2-4 ชั่วโมง)
- ระบุความสัมพันธ์ของการพึ่งพาเพื่อหลีกเลี่ยงการติดขัด
- จัดลำดับความสำคัญของงานในเส้นทางหลักก่อน
- อัปเดตสถานะงานเป็นประจำ

---

## `/write` - ดำเนินการเขียนบท

### หน้าที่

ตามรายการงานและแผน AI ช่วยในการสร้างสรรค์บท

### สถานการณ์การใช้งาน

- เมื่องานพร้อม
- ในแต่ละครั้งที่เขียน
- เมื่อต้องการให้ AI ช่วยสร้างเนื้อหา

### ผลลัพธ์

สร้างไฟล์ `stories/[story-name]/content/chapter-XX.md`

### วิธีการใช้

```
/write

# หรือระบุบท
/write บทที่ 5 - การพบกันครั้งแรก

# หรือตามงาน
/write [P0][ขึ้นอยู่กับ:1] เขียนบทที่ 1: อารัมภบท
```

### Skills ที่เปิดใช้งานอัตโนมัติขณะเขียน

- **Genre Knowledge**: ใช้รูปแบบตามประเภทเรื่อง
- **Dialogue Techniques**: ปรับปรุงฉากบทสนทนา
- **Scene Structure**: ควบคุมจังหวะของฉาก
- **Consistency Checker**: ตรวจสอบความสอดคล้องเบื้องหลัง

### แนวทางปฏิบัติที่ดีที่สุด

- เขียนครั้งละ 1-2 บท
- อ้างอิง specification และ plan
- ให้ AI สร้างกรอบก่อน แล้วค่อยเติมรายละเอียด
- บันทึกหลายเวอร์ชันเพื่อเปรียบเทียบและเลือก

---

## `/analyze` - วิเคราะห์ยืนยันคุณภาพ

### หน้าที่

ตรวจสอบคุณภาพและความสอดคล้องของงานอย่างครบถ้วน

### สองโหมด

1. **การวิเคราะห์กรอบ** (ก่อนเขียน)
   - ตรวจสอบความสมบูรณ์ของแผน
   - ตรวจสอบตรรกะ
   - ประเมินความเป็นไปได้

2. **การวิเคราะห์เนื้อหา** (หลังเขียน)
   - ตรวจสอบคุณภาพการดำเนินการ
   - ตรวจสอบความสอดคล้อง
   - ประเมินการบรรลุเป้าหมาย

### สถานการณ์การใช้งาน

- หลังจากเสร็จ 3 บทแรก (การตรวจสอบตั้งแต่เนิ่น ๆ)
- ทุก 5 บท (การตรวจสอบเป็นประจำ)
- หลังจากร่างเสร็จสมบูรณ์ (การตรวจสอบครบถ้วน)

### มิติการตรวจสอบ

```
/analyze
```

ตรวจสอบ:
- ✅ การปฏิบัติตามรัฐธรรมนูญ
- ✅ ความครบถ้วนตามข้อกำหนด
- ✅ ความสอดคล้องของเนื้อหา
- ✅ มาตรฐานคุณภาพ
- ✅ จังหวะและความตึงเครียด
- ✅ การพัฒนาเส้นทางตัวละคร

### แนวทางปฏิบัติที่ดีที่สุด

- รันตั้งแต่เนิ่น ๆ เพื่อพบปัญหาแต่เนิ่น ๆ
- อย่ารอให้เขียนเสร็จทั้งหมดแล้วค่อยวิเคราะห์
- ปรับแผนตามผลการวิเคราะห์
- บันทึกจุดที่ต้องปรับปรุง

## คำอธิบายคำสั่งติดตามโดยละเอียด

## `/track-init` - เริ่มต้นระบบติดตาม

### หน้าที่

สร้างไฟล์ JSON ที่จำเป็นสำหรับระบบติดตาม

### สถานการณ์การใช้งาน

- ครั้งแรกที่ใช้ฟังก์ชันติดตาม
- เมื่อเริ่มเรื่องใหม่

### ผลลัพธ์

สร้างใน `spec/tracking/`:
- `plot-tracker.json` - ติดตามเนื้อเรื่อง
- `timeline.json` - เส้นเวลา
- `character-state.json` - สถานะตัวละคร
- `relationships.json` - ความสัมพันธ์ตัวละคร

### ตัวอย่าง

```
/track-init
```

### แนวทางปฏิบัติที่ดีที่สุด

- รันก่อนเริ่มเขียน
- ต้องรันเพียงครั้งเดียว
- สามารถแก้ไขไฟล์ JSON ด้วยตนเอง

---

## `/track` - อัปเดตการติดตามแบบครบวงจร

### หน้าที่

อัปเดตข้อมูลการติดตามทั้งหมดอย่างชาญฉลาด

### สถานการณ์การใช้งาน

- หลังจากเขียนแต่ละบทเสร็จ
- หลังจากเกิดเหตุการณ์สำคัญ
- เมื่อต้องการจัดระเบียบข้อมูลเป็นระยะ

### กระบวนการ

1. AI อ่านบทล่าสุด
2. ดึงข้อมูลสำคัญ
3. อัปเดตไฟล์ติดตามทั้งหมด
4. ทำเครื่องหมายรายการที่ต้องยืนยันโดยมนุษย์

/track

- ตรวจสอบว่าการอัปเดตของ AI ถูกต้อง
- เพิ่มเติมรายละเอียดที่ AI พลาดด้วยตนเอง

## ข้อมูลอ้างอิงด่วน

### ขั้นตอนการทำงานขั้นต่ำ

```
/constitution → /specify → /write
```

### ขั้นตอนการทำงานมาตรฐาน

```
/constitution → /specify → /clarify → /plan → /tasks → /write → /analyze
```

### ขั้นตอนการทำงานครบวงจร

```
/constitution → /specify → /clarify → /plan → /tasks → 
/track-init → /write → /track → /analyze
```

### จังหวะที่แนะนำ

```
วันแรก: /constitution + /specify (2-3ชม.)
วันที่สอง: /clarify + /plan (2-3ชม.)
วันที่สาม: /tasks + เริ่ม /write (2-3ชม.)
วันที่ 4-10: /write (วันละ 1-2ชม.)
วันที่ 11: /analyze + วางแผนชุดถัดไป
```

## เทคนิคการผสมผสานคำสั่ง

### การทำซ้ำอย่างรวดเร็ว

```
1. /write บทที่ 1-3
2. /analyze การตรวจสอบตั้งแต่เนิ่น ๆ
3. ปรับ /plan ตามข้อเสนอแนะ
4. เขียนต่อด้วย /write
```

### การวางแผนเชิงลึก

```
1. /constitution (หลักการสำคัญ)
2. /specify (ข้อกำหนดโดยรวม)
3. /clarify (ขจัดความคลุมเครือ)
4. /plan (แผนรายละเอียด)
5. ใช้ /clarify หลายครั้งจนกว่าจะชัดเจนสมบูรณ์
6. เริ่ม /write
```

### การเขียนแบบสำรวจ

```
1. /constitution แบบย่อ
2. /specify แบบเบา
3. /write บทสำรวจโดยตรง
4. อัปเดต /specify และ /plan ตามผลลัพธ์
5. /write แบบมีโครงสร้าง
```
