## 1. เบื้องหลังการอัปเกรด

### 1.1 ปัญหาหลัก

**ข้อเสนอแนะจากผู้ใช้**:
> "Novel Writer ทำงานได้ดีสำหรับ 30 บทแรก แต่เมื่อเขียนถึงหลัง 30 บท AI เริ่มเสียสมาธิ ลืมการตั้งค่าและกฎเกณฑ์ก่อนหน้านี้"

**สาเหตุที่แท้จริง**:
- บริบทที่สะสมในนิยายยาวทำให้ AI กระจายความสนใจ
- ขาดกลไกการโหลดบริบทใหม่แบบบังคับ
- ไม่มีระบบฐานความรู้ที่มีโครงสร้าง
- การติดตามตัวละคร/โครงเรื่องต้องพึ่งพาความจำของมนุษย์

### 1.2 กลุ่มเป้าหมาย

- **ผู้ใช้หลัก**: นักเขียนมืออาชีพ สร้างสรรค์นิยายความยาวมากกว่า 500,000 คำ
- **ผู้ใช้รอง**: นักเขียนมือใหม่ที่ต้องการคำแนะนำเชิงระบบ
- **แผนอนาคต**: เตรียมโครงสร้างพื้นฐานสำหรับการพัฒนา VSCode Extension

## 2. การออกแบบแนวทางแก้ไข

### 2.1 การอัปเกรดสถาปัตยกรรมหลัก

#### ระบบฐานความรู้ (Knowledge Base)

**แนวคิดการออกแบบ**: สถาปัตยกรรมแบบแบ่งชั้น โหลดตามความต้องการ

```
.claude/knowledge-base/
├── README.md              # ตารางแผนที่คำสำคัญ + ดัชนีรวม
├── genres/                # ฐานความรู้ประเภทเรื่อง
│   ├── romance.md        # นิยายรัก (948 บรรทัด, ~27KB)
│   ├── mystery.md        # สืบสวนสอบสวน (1123 บรรทัด, ~31KB)
│   ├── historical.md     # อิงประวัติศาสตร์ (1127 บรรทัด, ~27KB)
│   ├── revenge.md        # 复仇 (846 บรรทัด, ~22KB)
│   └── wuxia.md          # 武俠 (143 บรรทัด, ~2.8KB)
├── craft/                 # เทคนิคการเขียน (สงวนไว้)
└── references/            # เอกสารอ้างอิง (สงวนไว้)
```

**ประสิทธิภาพของ Token**:
- แนวทางเดิม: 50 Skills × 40 tokens = ~2000 tokens
- แนวทางใหม่: 1 ตัวตรวจจับ (100 tokens) + ฐานความรู้ตามต้องการ (500 tokens) = 600 tokens
- **ประหยัดค่าใช้จ่าย token ได้ 75%**

**กลไกการเปิดใช้งานอัตโนมัติด้วยคำสำคัญ**:
```yaml
romance:
  keywords: [นิยายรัก, ความรัก, ความสัมพันธ์, โรแมนติก, เส้นทางความรัก, CP, เรื่องหวาน, เรื่องเศร้า]
  auto_load: genres/romance.md

mystery:
  keywords: [สืบสวน, สอบสวน, นักสืบ, คดี, ปริศนา, เบาะแส, ความจริง, ฆาตกร, อาชญากรรม]
  auto_load: genres/mystery.md
```

#### Quality Assurance Skills

**4 Skills หลัก**:

1. **setting-detector** (566 บรรทัด)
   - ฟังก์ชัน: ตรวจจับคำสำคัญในอินพุตผู้ใช้โดยอัตโนมัติ และโหลดฐานความรู้ที่เกี่ยวข้อง
   - โอกาสเปิดใช้งาน: เมื่อผู้ใช้พูดถึงประเภทเรื่อง การตั้งค่า พื้นหลัง
   - ตัวอย่าง:
     ```
     ผู้ใช้: "ฉันจะเขียนนิยายรักแนว复仇ในยุค 1920"
     ตัวตรวจจับระบุ:
       - "ยุค 1920" → china-1920s.md
       - "นิยายรัก" → romance.md
       - "复仇" → revenge.md
     โหลดฐานความรู้ 3 รายการโดยอัตโนมัติ
     ```

2. **pre-write-checklist** (541 บรรทัด)
   - ฟังก์ชัน: บังคับให้ AI ยืนยันว่าได้โหลดไฟล์หลัก 9 รายการก่อนรัน `/write`
   - แก้ปัญหา: **เป็นแนวทางแก้ไขหลักสำหรับปัญหาการเสียสมาธิหลัง 30 บท**
   - รายการตรวจสอบบังคับ:
     ```markdown
     ✓ 1. memory/constitution.md - รัฐธรรมนูญการเขียน
     ✓ 2. memory/style-reference.md - เอกสารอ้างอิงรูปแบบ
     ✓ 3. stories/*/specification.md - ข้อกำหนดเรื่อง
     ✓ 4. stories/*/creative-plan.md - แผนการเขียน
     ✓ 5. stories/*/tasks.md - งานปัจจุบัน
     ✓ 6. spec/tracking/character-state.json - สถานะตัวละคร
     ✓ 7. spec/tracking/relationships.json - เครือข่ายความสัมพันธ์
     ✓ 8. spec/tracking/plot-tracker.json - การติดตามเนื้อเรื่อง
     ✓ 9. spec/tracking/validation-rules.json - กฎการตรวจสอบ
     ```

3. **forgotten-elements** (147 บรรทัด)
   - ฟังก์ชัน: ตรวจสอบความถี่การปรากฏของตัวละคร/โครงเรื่อง/ปมเบื้องหลัง
   - เกณฑ์:
     - ตัวละคร: ไม่ปรากฏ 10 บท → เตือน
     - เส้นเรื่อง: ไม่มีความคืบหน้า 12 บท → เตือน
     - ปม: ไม่ถูกเก็บ 20 บท → เตือน
   - ตัวอย่างการเตือน:
     ```
     ⚠️ การเตือนตัวละคร:
     "ตัวละครรอง หลี่หมิง" ไม่ปรากฏ 10 บทแล้ว (ครั้งล่าสุด: บทที่ 5)
     - ควรจัดให้เขาปรากฏอีกครั้งหรือไม่?
     - หรือเรื่องราวของตัวละครนี้จบแล้ว?
     ```

4. **getting-started** (224 บรรทัด)
   - ฟังก์ชัน: แนะนำผู้ใช้ใหม่ แนะนำระเบียบวิธีเจ็ดขั้นตอน
   - เงื่อนไขการเปิดใช้งาน: ผู้ใช้พูดว่า "ฉันอยากเขียนนิยาย" "ใช้อย่างไร" เป็นต้น
   - เนื้อหา:
     - คำอธิบายกระบวนการ 7 ขั้นตอนโดยละเอียด
     - ตอบคำถามที่พบบ่อย
     - คำแนะนำสำหรับการใช้งานครั้งแรก

## 4. คำอธิบายเนื้อหาฐานความรู้โดยละเอียด

### 4.1 ฐานความรู้นิยายรัก (romance.md)

**เนื้อหาหลัก** (948 บรรทัด):

1. **การออกแบบส่วนโค้งความสัมพันธ์**
   - พบกัน → ชอบ → ทดสอบ → ยืนยัน → ความสัมพันธ์ลึกซึ้ง → วิกฤต → คืนดี → มั่นคง
   - เหตุการณ์สำคัญในแต่ละช่วง
   - เทคนิคการควบคุมจังหวะ

2. **จังหวะอารมณ์**
   - การออกแบบจังหวะหัวใจเต้นแรง (15+ ฉากคลาสสิก)
   - สูตรช่วงเวลาหวาน
   - การควบคุมระดับความหนักหน่วงของอารมณ์

3. **การสร้างเคมีความสัมพันธ์**
   - เทคนิคบทสนทนาที่คลุมเครือ
   - รายละเอียดภาษากาย
   - การออกแบบบทพูดคนเดียวภายใน

4. **การจัดการความขัดแย้ง**
   - การตรวจสอบความสมเหตุสมผลของความเข้าใจผิด
   - การใช้ตัวละครที่สามอย่างถูกต้อง
   - ความกดดันภายนอก vs ความขัดแย้งภายใน

### 4.2 ฐานความรู้สืบสวนสอบสวน (mystery.md)

**เนื้อหาหลัก** (1123 บรรทัด):

1. **หลักการเล่นที่ยุติธรรม (Fair Play)**
   - เบาะแสทั้งหมดต้องนำเสนอต่อผู้อ่าน
   - ไม่สามารถซ่อนข้อมูลสำคัญ
   - ความจริงต้องสามารถอนุมานได้

2. **การวางเบาะแส**
   - โครงสร้างสามชั้น: เบาะแสชัดเจน + เบาะแสซ่อนเร้น + ข้อมูลหลอก
   - ความหนาแน่นของเบาะแส: อย่างน้อย 3 เบาะแสที่มีประสิทธิภาพทุก 5 บท
   - การตรวจสอบย้อนหลัง: หลังจากเปิดเผยความจริงแล้วต้องสามารถอธิบายได้อย่างมีเหตุผล

3. **การจัดการผู้ต้องสงสัย**
   - กลุ่มผู้ต้องสงสัย 3-5 คน
   - ทุกคนมีแรงจูงใจ+โอกาส+ข้อสงสัย
   - การออกแบบจังหวะการตัดออก

4. **การเปิดเผยความจริง**
   - การเปิดเผยแบบโดมิโน vs การเปิดเผยครั้งเดียว
   - เงื่อนไขการพลิกผันที่ไม่คาดคิด
   - รายการตรวจสอบการเก็บปม

### 4.3 ฐานความรู้นิยายอิงประวัติศาสตร์ (historical.md)

**เนื้อหาหลัก** (1127 บรรทัด):

1. **เหตุการณ์สำคัญ vs รายละเอียดเล็กน้อย**
   - เหตุการณ์ประวัติศาสตร์สำคัญเป็นพื้นหลัง ไม่ใช่เส้นเรื่องหลัก
   - มุ่งเน้นที่ตัวละครในประวัติศาสตร์
   - หลีกเลี่ยงการเล่าประวัติศาสตร์แบบเรียงลำดับ

2. **เทคนิคการบรรยายแบบขาว-ดำ (Historical White Description)**
   - การบรรยายสภาพแวดล้อมที่简洁有力
   - ตัดคำคุณศัพท์สมัยใหม่
   - ตัวอย่างเปรียบเทียบ:
     ```
     ❌ ไม่ดี: "พระราชวังอันหรูหราโอ่อ่าตระการตาทำให้คนทึ่ง"
     ✅ ดี: "กำแพงสีแดง กระเบื้องสีเหลือง ชายคาโค้ง"
     ```

3. **การสร้างบรรยากาศยุคสมัย**
   - ความถูกต้องของเครื่องแต่งกาย มารยาท การเรียกขาน
   - การค้นคว้ารายละเอียดชีวิตประจำวัน
   - ความรู้สึกของภาษาในยุคนั้น

4. **ความสมดุลระหว่างความถูกต้องทางประวัติศาสตร์และความบันเทิง**
   - เหตุการณ์สำคัญต้องไม่บิดเบือน
   - รายละเอียดสามารถ虚构ได้อย่างสมเหตุสมผล
   - เรื่องราวของตัวละครเล็กมีพื้นที่ให้สร้างสรรค์มาก

### 4.4 ฐานความรู้复仇 (revenge.md)

**เนื้อหาหลัก** (846 บรรทัด):

1. **การสร้างความแค้นลึกซึ้ง**
   - ความแค้นต้องมีเหตุผลเพียงพอ
   - ต้อง铺垫อย่างละเอียดในระยะแรก
   - ผู้อ่านต้องเกิดความเห็นอกเห็นใจ

2. **จังหวะการตบหน้า**
   - ค่อยเป็นค่อยไป > ทำครั้งเดียว
   - เริ่มเล็กแล้วใหญ่ ค่อย ๆ เพิ่มระดับ
   - แต่ละครั้งต้องมีรูปแบบใหม่

3. **ความสมเหตุสมผลของการเติบโต**
   - พลังพิเศษต้องไม่เกินจริงเกินไป
   - การเติบโตต้องมี代价
   - หลีกเลี่ยงการ压倒โดยไม่ต้องใช้ความคิด

4. **ความสมดุลทางจริยธรรม**
   - การ复仇ต้องมีขอบเขต
   - อย่าทำร้ายคนที่ไม่ได้เกี่ยวข้อง
   - ความยุติธรรม vs ความสะใจ

### 4.5 ฐานความรู้นิยาย武俠 (wuxia.md)

**เนื้อหาหลัก** (143 บรรทัด, เวอร์ชันย่อ):

1. **กฎของยุทธภพ**
   - จิตวิญญาณของนักรบผู้กล้า
   - ระบบสมาคม
   - ความแค้นและความรัก

2. **ระบบวิชายุทธ**
   - พลังภายใน vs พลังภายนอก
   - เทคนิคการบรรยายท่าไม้ตาย
   - การกำหนดระดับความแข็งแกร่ง

3. **จิตวิญญาณของนักรบผู้กล้า**
   - "ผู้ยิ่งใหญ่ในยุทธภพ เพื่อชาติและประชาชน"
   - ความแค้นส่วนตัว vs หน้าที่อันยิ่งใหญ่
   - ระดับของความรวดเร็วและความแค้น

## 5. การเพิ่มประสิทธิภาพประสบการณ์ผู้ใช้

### 5.1 การปรับปรุงขั้นตอนการเขียน

**ก่อนหน้านี้ (v1.0.2)**:
```
ผู้ใช้รัน /write
  ↓
AI อ่านไฟล์เอง
  ↓
อาจพลาดไฟล์สำคัญ
  ↓
เสียสมาธิหลัง 30 บท
```

**ตอนนี้ (v1.0.3)**:
```
ผู้ใช้รัน /write
  ↓
pre-write-checklist เปิดใช้งานบังคับ
  ↓
AI ต้องระบุอย่างชัดเจนว่าได้อ่านไฟล์หลัก 9 รายการแล้ว
  ↓
ยืนยันว่าบริบทสมบูรณ์แล้วจึงเริ่มเขียน
  ↓
รักษาความสม่ำเสมอได้มากกว่า 50+ บท
```

### 5.2 การเปิดใช้งานฐานข้อมูลอัตโนมัติ

**ตัวอย่างสถานการณ์**:

ผู้ใช้ป้อน:
> "ฉันจะเขียนนิยาย复仇แนวรักในยุค 1920 ที่เซี่ยงไฮ้ นางเอกเป็นนักร้องสาว พระเอกเป็นหัวหน้าแก๊งชิงบัง"

**setting-detector ระบุอัตโนมัติ**:
- คำสำคัญ "ยุค 1920" + "เซี่ยงไฮ้" → พื้นหลังประวัติศาสตร์
- คำสำคัญ "复仇" → revenge.md
- คำสำคัญ "นิยายรัก" → romance.md
- คำสำคัญ "แก๊งชิงบัง" → อาจกระตุ้นความรู้เฉพาะยุค

**โหลดอัตโนมัติ**:
- `knowledge-base/genres/historical.md` (~1100 บรรทัด)
- `knowledge-base/genres/revenge.md` (~850 บรรทัด)
- `knowledge-base/genres/romance.md` (~950 บรรทัด)

**การใช้ Token ทั้งหมด**: ประมาณ 1500-1800 tokens

**คุณค่า**:
- AI ได้รับการสนับสนุนจากฐานความรู้เฉพาะทาง 3 รายการทันที
- ไม่ต้องให้ผู้ใช้ระบุเอง
- คุณภาพการเขียนเพิ่มขึ้นอย่างมีนัยสำคัญ

### 5.3 การแนะนำผู้ใช้ใหม่

**ก่อนหน้านี้**:
- ผู้ใช้ต้องเรียนรู้ระเบียบวิธีเจ็ดขั้นตอนด้วยตนเอง
- ไม่รู้ว่าจะเริ่มจากตรงไหน
- มักข้ามขั้นตอนหรือพลาด

**ตอนนี้**:
- พูดว่า "ฉันอยากเขียนนิยาย" → getting-started เปิดใช้งานอัตโนมัติ
- ให้คำอธิบายกระบวนการทั้งหมด
- ตอบคำถามที่พบบ่อย
- แนะนำให้เริ่มจากโปรเจกต์ง่าย ๆ

## 7. การปรับปรุงประสิทธิภาพ

### 7.1 ประสิทธิภาพการใช้ Token

**การวิเคราะห์เปรียบเทียบ**:

| แนวทาง | การใช้ Token | ขอบเขตการครอบคลุม |
|------|-----------|---------|
| แนวทางเดิม (50 Skills แยก) | ~2000 tokens | 50 จุดความรู้ |
| แนวทางใหม่ (ฐานความรู้แบบแบ่งชั้น) | ~600 tokens | โหลดตามความต้องการ |
| **อัตราการประหยัด** | **75%** | **ยืดหยุ่นกว่า** |

### 7.2 ความสามารถในการขยาย

**ต้นทุนในการเพิ่มฐานความรู้ใหม่**:
1. สร้างไฟล์ใหม่ใน `knowledge-base/genres/`
2. เพิ่มการจับคู่คำสำคัญใน `README.md`
3. ไม่ต้องแก้ไขโค้ดใด ๆ

**ตัวอย่าง**: การเพิ่มฐานความรู้ "นิยายวิทยาศาสตร์"
```yaml
# knowledge-base/README.md 新增:
scifi:
  keywords: [นิยายวิทยาศาสตร์, SF, อวกาศ, อนาคต, หุ่นยนต์, AI, ไซเบอร์พังค์]
  auto_load: genres/scifi.md
```
